### What are servlets and servlet containers

### Servlet

A **Servlet** is a Java class that runs on a web server and handles HTTP requests and responses.

Think of it as the Java equivalent of a controller that:

1. Receives a request from a browser/app.
2. Processes the request.
3. Generates a response.
4. Sends the response back to the client.

Example:

```
@WebServlet("/hello")
public class HelloServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
                         throws IOException {

        response.getWriter().println("Hello World");
    }
}
```

When a user visits:

```
http://localhost:8080/hello
```

the servlet's `doGet()` method executes and returns:

```
Hello World
```

---

### Servlet Container

A **Servlet Container** (also called a **Web Container**) is the software that manages servlets.

Examples:

- Apache Tomcat
- Jetty
- Undertow

Without a servlet container, a servlet is just a Java class sitting in a folder.

The container is responsible for:

#### 1. Creating servlet objects

You never do:

```
new HelloServlet();
```

The container does it.

---

#### 2. Managing the servlet lifecycle

```
Load Servlet
    ↓
init()
    ↓
service()
    ↓
destroy()
```

Example:

```
public void init() {
    // called once when servlet starts
}

protected void doGet() {
    // called for every request
}

public void destroy() {
    // called before shutdown
}
```

---

#### 3. Listening on HTTP ports

The container listens on:

```
localhost:8080
```

When a request arrives:

```
Browser   ↓Tomcat   ↓Servlet
```

---

#### 4. Converting HTTP data into Java objects

Instead of parsing raw HTTP:

```
GET /hello HTTP/1.1
Host: localhost
```

you get:

```
HttpServletRequest request
HttpServletResponse response
```

which are much easier to use.

---

#### 5. Thread management

Suppose 100 users access:

```
/hello
```

The container creates/manages threads and invokes the servlet safely.

You don't manually create threads for each request.

---

### In Spring Boot

When you run:

```
@SpringBootApplication
public class App {
    public static void main(String[] args) {
        SpringApplication.run(App.class, args);
    }
}
```

Spring Boot automatically starts an embedded servlet container:

- Spring Boot + MVC → usually embedded Apache Tomcat
- Can also use Jetty or Undertow

Your controller:

```
@RestController
public class UserController {

    @GetMapping("/users")
    public String users() {
        return "Users";
    }
}
```

is ultimately executed through the servlet infrastructure.

Internally the flow is:

```
Browser
   ↓
Tomcat (Servlet Container)
   ↓
DispatcherServlet
   ↓
Controller
   ↓
Response
```

---

### DispatcherServlet

In Spring MVC, the most important servlet is:

```
DispatcherServlet
```

It is a servlet created by Spring.

Flow:

```
Request
   ↓
DispatcherServlet
   ↓
Controller
   ↓
Service
   ↓
Repository
   ↓
Response
```

This is why when learning Spring MVC, understanding servlets helps a lot—the entire framework is built on top of the Servlet API.

### Simple Analogy

Imagine a restaurant:

- **Customer** → Browser
- **Waiter** → Servlet Container
- **Chef** → Servlet

The waiter receives orders, manages tables, and brings food. The chef only focuses on preparing the food.

Similarly:

- Servlet focuses on business logic.
- Servlet Container handles networking, lifecycle, threads, and request management.
